# v2.0.4 — In Development

## Villager Trading

| Tag | Change |
|---:|---|
| ![feature] | <details><summary>Stock-market-driven villager trade repricing</summary><br>Villager and wandering-trader offers for market-listed items are rewritten at interaction time: emerald costs are replaced by the configured trading-currency item, priced from the live stock market with configurable margins (villager buys below market, sells above market). Items without a market keep their normal vanilla emerald trades — offers convert automatically when a market is created and revert when it is deleted.<br><br><ul><li>New <code>VillagerTrading</code> group in <code>settings.json</code>: <code>ENABLED</code> (default <code>true</code>), <code>PRICE_REFRESH_INTERVAL_MINUTES</code> (default <code>20</code> ≈ one Minecraft day), <code>VILLAGER_BUY_MARGIN</code> (default <code>0.8</code>), <code>VILLAGER_SELL_MARGIN</code> (default <code>1.2</code>).</li><li><b>Master-authoritative:</b> only the master server reads the settings; it broadcasts the price table (including the currency item) to all slave servers on a new server-to-server packet, so enabling it on the master enables it everywhere.</li><li>Original emerald offers are stored in world data, so disabling the feature restores every villager's original trades. Item ingredient slots (e.g. book + emeralds for enchanted books) are preserved — only the emerald part becomes currency.</li></ul></details> |
| ![feature] | <details><summary>Pay merchant money costs with any combination of denominations</summary><br>Offers priced in BankSystem money are matched by <b>value</b>, not by exact note type: a 20.00 cost can be paid with 1×20, 2×10, 4×5, or even a single 50 note, split across both payment slots.<br><br><ul><li>Exact combinations are consumed cleanly; overpayment returns exact change (fewest notes) into the payment slots, the inventory, or dropped at the player's feet — no value is ever lost, even when shift-clicking through a whole stack of large notes.</li><li>Clicking an offer row auto-fills the payment slots with money from the inventory, even when the exact denomination shown on the offer is not owned.</li><li>Only genuine BankSystem money is matched by value — renamed/modified notes, other mods' currencies, and non-money offers keep untouched vanilla behavior.</li></ul></details> |

## Performance

| Tag | Change |
|---:|---|
| ![improvement] | <details><summary>Massively reduced CPU usage of the default orderbook volume distribution</summary><br>The plugin that keeps orderbooks filled with baseline volume no longer recomputes its full price distribution for every market on every update — a hot spot that previously ran ~1000 expensive per-price-level calculations per market, per cycle.<br><br><ul><li>Each market's target distribution is now cached and only recomputed when something actually changes: the market price, the visible price range, or the plugin settings (volume scale / default price).</li><li>When several markets do need a refresh, they take turns round-robin — at most one market recomputes its distribution per update cycle, so servers with many markets no longer pay the full cost all at once. Market resets, settings changes, and price-range shifts still refresh immediately.</li><li>The per-price-level computation itself is cheaper: loop-invariant values (current market price, currency scale factor) are fetched once per range instead of once per price level.</li><li><b>Plugin API:</b> new <code>IPluginOrderBook.getDefaultRawVolume(backendStartPrice, count)</code> batch method lets plugins sample a contiguous price range in one cheap call instead of ~1000 single-price calls.</li></ul>While prices are static the heavy math does not run at all; only a lightweight convergence loop remains per update.</details> |

## Plugins

| Tag | Change |
|---:|---|
| ![feature] | <details><summary>VolatilityPlugin: player trading now permanently shifts market prices (economy self-balancing)</summary><br>The plugin's random walk no longer anchors on the static default price — it follows a <b>price equilibrium</b> driven by cumulative player trading. Sustained net player selling lowers a market's price range; net buying raises it. The shift is persistent and self-balancing: heavily farmed items get cheaper, heavily consumed items get more expensive.<br><br><ul><li>The impact scales with the item's <b>Abundance</b> — common items absorb more trading before their price moves.</li><li>The exponential impact curve means the price approaches but can never reach 0; additionally, the equilibrium is clamped between configurable min/max multiples of the default price.</li><li>Four new per-market GUI-editable settings: <b>Player Flow Influence</b> (on by default), <b>Flow Sensitivity</b> (default <code>500</code>), <b>Min Price Multiplier</b> (default <code>0.05</code>), <b>Max Price Multiplier</b> (default <code>20</code>).</li><li>Disable the Player Flow Influence toggle or reset the market's <b>Net Flow</b> to return to the old anchored-on-default-price behavior.</li><li>Existing worlds keep working — old saved plugin settings load with the new defaults.</li></ul></details> |

## Administration

| Tag | Change |
|---:|---|
| ![feature] | <details><summary>In-game Mod Settings screen for server admins (master server only)</summary><br>A new <b>"Mod Settings"</b> button in the management screen (shown only on the master server, requires op level 2) opens an editor for every <code>settings.json</code> setting — no more manual file editing.<br><br><ul><li>Settings are grouped like in the file; every input is validated and clamped server-side, so invalid values can never be saved.</li><li>The trading currency is selected via an item picker instead of typing item IDs.</li><li>Settings that only take effect after a server restart (virtual orderbook size, trading currency) are marked with a <b>"⟳ Restart required"</b> label and tooltip.</li><li>Villager-trading changes propagate to connected slave servers immediately.</li><li>Changes are saved to <code>settings.json</code> right away; <b>Defaults</b> and <b>Reload</b> buttons restore default or currently saved values.</li></ul></details> |

## News Event System

| Tag | Change |
|---:|---|
| ![feature] | <details><summary>Craftable newspaper and market-moving news events</summary><br>A new <b>NewsPlugin</b> randomly publishes news events — headlines that temporarily (or permanently) move the prices of the affected markets. Players read the feed through a craftable <b>newspaper item</b>, also openable from the Trade and Management screens and via <code>/stockmarket news</code>.<br><br><ul><li>Events are defined in JSON files under <code>config/StockMarket/news/</code>; eight default events ship out of the box, and a documented schema lets admins author their own.</li><li>Each event has a category (shock, trend, crash, permanent shift, …), a per-market impact envelope, and a headline/body shown in the newspaper.</li><li>An <b>opt-in popup toast</b> notifies interested players the moment news breaks; players who were offline get a catch-up toast on join.</li><li>The plugin exposes per-market settings (news enabled, sensitivity) and multiplies the volatility target price in a separate finalize stage, so headlines translate into real, bot-driven price movement regardless of plugin order.</li></ul></details> |
| ![feature] | <details><summary>Per-event black &amp; white pictures</summary><br>Every news event can carry its own picture, shown in the newspaper, the event details screen, and as a thumbnail in the admin Active-events tab.<br><br><ul><li>Pictures are grayscale PNGs (16..512 px, square, max 128 KiB) loaded from <code>config/StockMarket/news/pictures/</code>; baked defaults can be bundled in the jar, and events without art fall back to a deterministic procedural placeholder.</li><li>Published records snapshot the picture by hash, so the exact art a player saw is preserved in history even if the source file later changes.</li><li>Clients fetch pictures on demand in hash-batched, per-player rate-limited requests and cache the decoded textures with lifecycle cleanup on disconnect.</li></ul></details> |
| ![feature] | <details><summary>Multi-step impact sequences</summary><br>Events can define a <code>sequences[]</code> of phases instead of a single impact — for example a spike that decays into a slump and then recovers. Each phase has its own duration and per-market envelope, normalized at parse time. The plugin runs the phases in order, and admins can <b>skip the current phase</b> or <b>hard-stop</b> a running event (including its recovery phase) from the management window.</details> |
| ![feature] | <details><summary>Trigger requirements, world-event registry and registry commands</summary><br>Events can declare <b>requirements</b> that must hold before they are eligible to fire. A persistent <b>world-event registry</b> records which named events have occurred in the world, and a predicate engine evaluates requirements against it at fire time. Admins can inspect and clear the registry through the <code>/stockmarket news</code> registry subcommands.</details> |
| ![feature] | <details><summary>Event chains</summary><br>A news event can schedule follow-up events when it fires, forming a chain (e.g. a scandal that triggers a later recovery rally). Chained events pass through the same eligibility and requirement checks as randomly scheduled ones.</details> |
| ![feature] | <details><summary>News plugin management window and admin commands</summary><br>The NewsPlugin has a dedicated management window: an <b>Active</b> tab (running events with per-market impact, picture thumbnail, and per-event trigger / stop / skip-phase controls), an event list with a details popup and per-event enable/disable, scheduler-frequency controls, and an upcoming-events timeline with ETAs.<br><br>The <code>/stockmarket news</code> command opens the newspaper for players; admin subcommands <code>reload</code>, <code>trigger</code>, <code>list</code>, <code>info</code> and <code>stop</code> (plus registry list/clear) manage the system from chat.</details> |
